Service 2.1 · KOL Identification & Mapping

Pipeline & report guide

Two guides in one page. Tab 1 explains how the pipeline finds and ranks the true opinion leaders for an indication. Tab 2 explains how to read the final HTML report — what every number, colour and chart means — written so anyone can follow it, technical or not.

In one sentence

For an indication (e.g. Obesity) we find, per doctor, the sources that genuinely show that doctor engaging with the topic — web statements they actually made and PubMed articles they actually authored on-topic — then rank doctors by a composite score that blends three things: how much verified evidence they have, how widely they collaborate, and how focused their output is on the topic. Everything that is merely a keyword co-occurrence is thrown away before scoring.

Why the design looks the way it does

The very first version scored doctors with 0.45·pub + 0.30·cf + 0.25·digi and produced professionally wrong rankings — human geneticists and lab-medicine physicians landed in the Obesity top-50. Two failure modes caused it, and the whole design exists to prevent them from ever coming back:

1 · Co-occurrence ≠ engagement

The old cf_count only meant an indication keyword and a doctor's name appeared in the same document. It never checked whether the doctor actually said anything about the topic.

2 · Wrong signals dominated

A "DigiScore" (25% weight) floated doctors with ≤1 publication to the top, and PubMed "relevance" was a global similarity cut with no per-doctor topical check.

The one principle that never bends

Relevance is decided one source at a time, by an LLM, and only sources that pass count. "Genuine" is decided the same way on both evidence tracks: an LLM confirms the doctor is actively engaging with / sharing a view on the indication in that source. The composite score's other two ingredients (reach and focus) are computed on top of that verified evidence — they can re-rank doctors who already cleared the bar, but they can never let an unverified doctor in. No DigiScore. No raw keyword counts. No global similarity percentile.

The cost funnel — cheap first, LLM last

We must not send thousands of documents to the expensive LLM. So we narrow the field with cheap database queries (plus a fast local embedding step) first, and only the survivors ever reach the LLM. This is the heart of the design:

ALL doctors (~1,176)Stage 01 · cheap SQL + a local embedding call, no LLM · count candidate web + PubMed sources per doctor
▼
Top 100 candidatesranked by candidate-source count (an upper bound on the real score) · everyone else dropped before any LLM spend
▼
Fetch full textStage 02 · web CONTENT + PubMed title/abstract for the 100
▼
LLM verifiesStage 03 · ingest → ground → verify · a source counts only if a claim survives both checks
▼
KOLs & Rising StarsStage 04 · composite score → tenure buckets + absolute floors → tiers · Stage 05 reports the top 25

The 100 → 25 buffer absorbs candidates that look strong on cheap counts but collapse under verification — exactly the off-target-specialty problem. They simply fall out; a genuine doctor takes the slot. No hardcoded specialty exclusions — relevance verification does the filtering. (The 100 is top_n_candidates, set manually today — see Open questions at the end of this tab.)

Two evidence tracks

Every source is one of two kinds. Both feed the same score in the end, but they are gathered and verified slightly differently:

Web / practice communication

From the LLM_VALIDATION table. Here the doctor's authorship is unknown, so attribution is the whole problem — the LLM must confirm the named doctor is actively discussing the indication, not merely co-present on the page.

Candidate net = a cheap keyword gate (NEAR_BY=1, IS_OLD=0, IS_DOCTOR=1, IN_RELATION>29, keyword ∈ indication terms) combined with a vector-search arm that recovers on-topic pages the keyword gate would miss (see below).

PubMed science

From PUBMED_ARTICLE_MAPPING (MERGE_RESULT>1 = verified author) → text from CORE.PUBMED.ARTICLE. Authorship is already proven, so the LLM's job is relevance: is this article genuinely about the indication? Co-authors (incl. doctors outside our targeting) come structurally from CORE.PUBMED.AUTHOR.

New: the vector-search arm on web (hybrid recall)

The keyword gate is precise but literal: it can miss a doctor whose web page discusses obesity without ever containing an exact indication term (synonyms, paraphrase, German/English mix). The vector arm recovers those. It is turned on by [hybrid].hybrid_relevance = true.

  • Embed the query: the indication name + its PCA terms are turned into a 768-number "meaning vector" using a local ONNX model (VectorCreator, GTE multilingual, L2-normalised) — no external API, runs in-process.
  • Match by meaning: that vector is cosine-compared against pre-computed page embeddings in WEBSITES_VERTICAL_EMBEDDINGS_512. A page counts as a candidate if similarity clears vector_sim_threshold (default 0.55), keeping the top vector_top_k_per_hcp (default 20) per doctor, still gated by NEAR_BY/IS_DOCTOR.
  • Union, not replacement: keyword candidates ∪ vector candidates are merged (merge_web_ids). The vector arm only ever widens who gets a chance — it never bypasses Stage 03.

Because Stage 03's LLM verify remains the sole precision arbiter, a wider net cannot inflate anyone's final score — it can only give a genuine-but-keyword-invisible doctor the chance to earn one. A PubMed vector arm and a cross-encoder Reranker are vendored and config-gated but off by default and unwired this iteration.

The six stages

Each stage writes a JSON checkpoint and is resume-safe (skips if its output exists, unless you pass --force). Stages 01–05 run in order; Stage 06 is an optional backtest.

01

Fetch & shortlist

01_fetch_and_shortlist.py
What happens (cheap SQL + local embeddings, no LLM):
  • Load the indication's terms — CONTENT_FRAME_SPEC where PCA='X'.
  • Web candidates per doctor: keyword gate ∪ vector-recall arm (see the hybrid box above), merged into one candidate set + a count.
  • PubMed candidates per doctor: verified authorship (MERGE_RESULT>1) ∩ a topical CF-flag, within a 10-year window (pubmed_window_years), CF-treffer-weighted. The window is anchored on the anchor year (see below), not today's date.
  • Publication-history bars (display only): two separate queries count each doctor's PubMed articles per year over 10 years (pub_history_years): one keeps the CF/topic filter (the indication-relevant subset), the other removes it (total_pub_by_year = the doctor's total output per year). Together they drive the report's total-vs-relevant bar chart and Stage 04's score-development reconstruction — they are never sent to the LLM and never affect the score.
  • Shortlist: candidate_score = web_candidate_count + pubmed_candidate_count (an upper bound on the final verified relevance) → flag the top 100 (top_n_candidates) as shortlisted. Metadata (name, city, specialty, A/B/C/D rating) is attached; specialty is displayed, never used to filter. Also builds a per-doctor total-source count used later as the "focus ratio" denominator.

03

LLM wiki-build — where relevance is verified

03_wiki_build.py
What happens, per source (web & PubMed alike):
  • Ingest (LLM #1) — extract whether the doctor is genuinely engaging with the indication: a verbatim supporting quote, a one-line statement, sentiment, the themes (which indication terms), and any other doctors mentioned.
  • Quote-grounding (deterministic, no LLM) — the verbatim quote must literally appear in the source text, or the claim is dropped for free — before spending a verify call.
  • Verify (LLM #2, a different, adversarial model) — returns true only if the source really shows this doctor engaging with the indication.
  • Map — resolve the doctor and any mentioned doctors against our targeting roster by name.
A source counts as relevant for a doctor only if it yields ≥1 grounded + verified claim. This stage is untouched by the hybrid arm and the composite score — it stays the sole precision arbiter. Besides data/wiki.json, it writes a browsable wiki/<ts>/ tree of what was ingested and verified. It also records, per verified PubMed claim, the article's year (feeding the score-development reconstruction), and threads through the total-publications-per-year map from Stage 01.

→ data/wiki.json  (+ wiki/<ts>/ tree)
04

Assemble the ranking — composite score, tenure buckets, floors

04_assemble_kols.py
What happens (no LLM): compute three factors per doctor and combine them into one kol_score, then sort doctors into two mutually-exclusive buckets and gate the "KOL" label with absolute floors:
  • Relevance — count of verified web + verified PubMed sources (the backbone). Normalised across the pool.
  • Reach — distinct co-authors across the doctor's verified-relevant PubMed articles (breadth of collaboration). Normalised across the pool.
  • Ratio (focus) — verified-relevant sources ÷ all of the doctor's sources. Used raw (not normalised) — it is already an intrinsic 0–1 quantity and indication-independent.
  • Tenure partition first: a doctor is a Rising Star if their relevant tenure (years since their first verified-relevant PubMed year) is short — ≤ rising_star_max_tenure_years (3) — and they have ≥ rising_star_min_pubs (3) verified-relevant articles. Rising stars are pulled out first and can never also be KOLs.
  • Everyone else is KOL-eligible, but only becomes an actual KOL if it clears four absolute floors (min verified sources, min focus ratio, recent activity, min co-author reach). This is the "cap at the bottom" that gives the label meaning regardless of the pool — see Tab 2 and the Open questions.
  • Tiers A B C are percentile bands computed over the KOL pool only (rising stars no longer distort the distribution). A rising star whose score already reaches Tier-A level is flagged Breakout.
  • Also builds each KOL's / rising star's score-development trajectory (year-by-year composite replay), themes, the co-author network, and web co-mentions.
The exact formula, floors, tiers, buckets and the trajectory chart are explained in Tab 2.
CORE.PUBMED.AUTHOR no LLM — plain arithmetic on Stage-03 output

The anchor year (time machine)

PubMed sources have dates; web sources do not. To keep runs reproducible and to enable the backtest, PubMed windows are anchored on [funnel].as_of_year:

  • as_of_year = latest → anchor = the most recent year present in the PubMed CF table.
  • as_of_year = 2018 (a pinned year) → the PubMed scoring window, history window and focus-ratio denominator are all capped at 2018, so a past year can be replayed. Web sources are timestamp-free and always shown as-is.

When a run is pinned to a past year, the report shows a "Backtest view" banner so nobody mistakes it for a live ranking.

Under the hood

  • LLMs: AWS Bedrock (eu-central-1) — Amazon Nova Pro for ingest, Qwen3-235B for the adversarial verify. Temperature 0.
  • Embeddings: a local ONNX model (GTE multilingual, 768-dim) for the vector-recall arm — no external API. The cross-encoder Reranker is vendored but not yet wired in.
  • Snowflake auth: AWS Secrets Manager private-key auth via the shared ParameterManager / SecretReader helpers.
  • Cost control: only the 100 shortlisted doctors' sources within the anchored 10-year window reach the LLM; the 10-year publication history is cheap SQL counts for the charts and never hits the LLM. Grounding drops fabricated quotes before any verify call is spent.
To confirm on the first live run: the CF-flag column layout in PUBMED_CONTENT_FRAME_SINGLE_TBL; that IN_RELATION is numeric so >29 is the right cut; that IS_OLD=0 is the intended web freshness filter; the real vector width of the _512 embeddings column (the pipeline currently queries at 768 dims); and the fully-qualified locations of the cross-database tables.

Open discussions & open questions

Decisions we've made deliberately but keep open for revision, plus questions we still want answered — written down so the trade-offs aren't lost between iterations.

1 · Norm-based scoring with a floor at the bottom Decided — values still open

Today the scores are calculated only from the run's own data — there is no universal source of truth for who is a KOL. That felt strange, so we weighed the two ways to define "KOL":

Universal / absolute thresholds

+ "KOL" means the same thing everywhere; runs are directly comparable across indications.
– A threshold tuned on a common, data-rich indication (e.g. Obesity) is simply wrong for a rare one (few HCPs, thin PubMed) — you'd crown nobody, or everybody. Brittle; needs re-tuning per indication.

Pure norms (relative to the pool)

+ Self-calibrating — flexes to any indication automatically, rare or common, with no re-tuning.
– No absolute meaning: the top of a weak pool is still labelled "KOL", and a 0.9 in one run isn't a 0.9 in another.

What we chose: keep the ranking and tiers norm-based (so it flexes across indications), but cap the bottom with four absolute floors (min verified sources, min focus ratio, recent activity, min co-author reach) so a doctor must clear a real engagement bar before "KOL" means anything. We get the flexibility of norms and the meaningfulness of a hard minimum. Still open: are the floor values (5 verified / 0.10 ratio / 5y / 3 co-authors) the right ones, and should the floors themselves scale with indication prevalence or the client's HCP universe rather than being fixed?

2 · Should Rising Stars also have a composite-score ceiling? Open

Right now the Rising bucket is decided only on activity + tenure: first verified-relevant PubMed year within the last 3 years and ≥ 3 verified-relevant articles. There is no cap on the composite score.

Should we add one — i.e. a doctor is "Rising" only if their composite is below some level (say below the Tier-A threshold, or a fixed 0.60) — so a genuinely high-scoring newcomer is called arrived, not rising? We already flag such doctors with a Breakout badge; the question is whether that's enough, or whether a hard ceiling should move them out of the Rising bucket entirely. Trade-off: a score ceiling re-introduces a scored blend into what is currently a clean, easy-to-explain tenure partition.

3 · What fraction of HCPs actually end up as KOLs? → auto-size the candidate cut Open — needs data

The funnel currently keeps a fixed top 100 candidates (top_n_candidates), set manually. But if the model is meant to be flexible and movable, the candidate cut should scale to the client's HCP universe — a fixed 100 under-serves a large panel and over-serves a small one.

To size it automatically we need the empirical KOL yield rate: across past targetings, what fraction of the client's supplied HCPs end up as KOLs at the end? With that, roughly
top_n_candidates ≈ (expected_KOL_rate × total_HCPs) ÷ funnel_survival_rate.
Open questions: what is that rate from our historical runs, is it stable or strongly indication-dependent, and what's the shortlist→KOL survival rate through Stages 03–04 that we'd divide by?

4 · Smaller items to revisit

  • Window lengths. We moved the PubMed scoring window and the display history both to 10 years (from 5 and 20 respectively). Is 10 right for a fast-moving field vs. a slow one — should the window itself be indication-aware?
  • Tier percentiles. Tiers A/B (85 / 60) are percentiles over the KOL pool. Should Tier A specifically also carry an absolute floor, so "Tier A" is comparable across runs the way the KOL floor made "KOL" comparable?
  • Web recency. Web sources are timestamp-free, so they get no recency weighting and are frozen across backtest years. Acceptable, or do we want a web-recency proxy?
  • Seniority. Reach measures co-author breadth, not author position / seniority — there's no author-order column in the PubMed data today. Worth sourcing?
  • Dead config key. rising_star_growth is left over from the old "accelerating" rule and is no longer read since the tenure rewrite — remove it to avoid confusion.
  • No ground truth. There is no supervised "is-a-KOL" label, so the only empirical validation is the Stage 06 time-machine backtest. Is that enough evidence for the client, or do we want a human-labelled spot-check set?

The composite score — the single most important number

Every KOL has one headline number, the composite score (kol_score), roughly between 0 and 1. It is a weighted blend of three factors. Think of it as: "how much verified evidence (60%), how widely they collaborate (25%), and how focused they are on the topic (15%)."

kol_score = 0.60 × norm_relevance + 0.25 × norm_reach + 0.15 × ratio(raw)

Relevance and reach are normalised across the pool (a doctor's standing relative to the other candidates). Ratio is the exception: it is used raw, because it is already an intrinsic 0–1 focus measure that means the same thing in a common indication or a rare one — normalising it would throw that absolute meaning away.

Relevance weight 0.60

How many sources an LLM verified as genuinely on-topic for this doctor (web + PubMed). This is the backbone — the thing we actually set out to measure.

Reach weight 0.25

How many distinct co-authors the doctor has across their verified PubMed articles (de-duplicated by ORCID or name; the doctor themselves excluded). A proxy for influence breadth.

Ratio (focus) weight 0.15 · raw

Verified-relevant sources ÷ all of the doctor's sources. High = the doctor is a topic specialist; low = the topic is a small part of a broad output. Neutral (0) for very thin profiles so they're neither punished nor rewarded. Used raw, not normalised.

Weight vs. Norm vs. Contribution — the three columns you'll see

Click "Composite … — how it was scored" on any KOL row (or open a profile card) and you get a small table with exactly these columns. Here is what each one means, in order:

ColumnPlain meaningRange

NormFor relevance and reach: this doctor's normalised value — their standing relative to the other shortlisted doctors, squeezed onto a 0–1 scale. 1.0 ≈ best in the pool, 0.0 ≈ lowest. Raw counts are not used directly; ranking-within-the-pool is, so one runaway doctor can't dominate. For ratio the "Norm" column just shows its raw 0–1 value (ratio is never normalised).0 → 1

So "why is this doctor high?" is answerable exactly: 0.54 of their 0.77 comes from verified relevance — the evidence is doing most of the work, with reach and focus fine-tuning the order.

How "Norm" is actually computed

The default method is percentile (config normalization; minmax and zscore also available), and it applies to relevance and reach only — ratio is used raw. Percentile means: for each of those two factors, a doctor's norm is essentially the fraction of the pool scoring below them, rescaled so the pool's best = 1 and worst = 0, with ties sharing a rank. The key consequence for non-technical readers: those norms are always relative to this run's shortlist — a 0.9 means "near the top of these 100 candidates", not an absolute quality score. This relativity is exactly why the "KOL" label also needs the absolute floors below.

KOL vs Rising Star — two separate buckets

A doctor is sorted into one of two mutually-exclusive buckets — never both. The split is a pure tenure partition, decided before any tier:

  • Rising Star — their relevant tenure (years since their first verified-relevant PubMed publication) is short: ≤ rising_star_max_tenure_years (3) — and they have at least rising_star_min_pubs (3) verified-relevant articles (so it's a genuine climb, not a one-off). Rising stars are taken out first and can never also be KOLs. A doctor with no verified PubMed year at all has undefined tenure and is never a rising star.
  • KOL-eligible — everyone else: long relevant tenure, or web-only (no publication tenure), or too few verified pubs to count as rising.

This is a deliberate trade-off: a prolific but very recent newcomer is bucketed as Rising, not KOL. If such a doctor's composite already reaches Tier-A level they get a Breakout badge, so an exceptional fast riser is still surfaced. (Whether the rising bucket should also have a score ceiling is an open question — see the end of Tab 1.)

What makes a KOL a KOL — the absolute floors

Tiers are relative (percentiles of the pool). On their own, relative bands would crown "KOLs" even in a weak pool — the top of a thin field is still just the top of a thin field. So a KOL-eligible doctor is only labelled an actual KOL if it clears four absolute floors — the same fixed bars in every run, independent of the pool. Fail any one, and the doctor lands in neither bucket:

FloorConfigMeaning
Min verified sourceskol_floor_min_verified = 5≥ 5 verified web + PubMed sources. Applies to everyone, web-only doctors included.
Min focus ratiokol_floor_min_ratio = 0.10At least 10% of the doctor's output is on-topic. A neutral (thin-denominator) ratio fails this.
Recent activitykol_floor_active_within_yrs = 5Any verified web source (timestamp-free, treated as current) or a verified PubMed pub within 5 years of the anchor year.
Min co-author reachkol_floor_min_coauthors = 3≥ 3 distinct co-authors — waived for a purely web-based voice with no PubMed at all, so it isn't silently dropped.

Why norms + a bottom cap, and not universal thresholds? Keeping the ranking norm-based lets the model flex across a common indication and a rare one without re-tuning — "top of this pool" travels. Adding absolute floors at the bottom restores the one thing pure norms lack: a doctor must clear a real engagement bar before "KOL" means anything. We get the flexibility of norms and the meaningfulness of a hard minimum. The tuning of the floor values is an open question — see the end of Tab 1.

How tiers are divided

Tiers are bands of the composite-score distribution over the KOL pool only (doctors with is_kol == True; config tier_a_percentile = 85, tier_b_percentile = 60). Rising stars are excluded from this distribution, so they no longer distort where the bands fall:

  • A — score in roughly the top 15% of the KOL pool (at/above the 85th percentile).
  • B — score between roughly the 60th and 85th percentile (the next ~25%).
  • C — the remaining KOLs (the bottom ~60% of the pool).

Tiers are relative to this run's KOL pool, not fixed thresholds — which is exactly why the absolute floors above decide who is a KOL in the first place. Doctors who aren't KOLs (rising stars, and anyone failing a floor) carry no tier. Doctors with zero verified sources are dropped entirely.

How rising stars are calculated

The Rising bucket is a tenure test, computed purely from verified-relevant PubMed activity (never from unverified keyword counts, so an off-topic publication burst can't crown anyone). A doctor is Rising when both hold:

  • Short relevant tenure — their first verified-relevant PubMed year is within the last rising_star_max_tenure_years (3) years of the anchor year. They haven't been on the topic long.
  • Genuinely active — at least rising_star_min_pubs (3) verified-relevant articles, so it's a real climb rather than a single paper.

Because the buckets are mutually exclusive, nobody in the Rising list appears in the KOL ranking. A rising star whose composite already reaches the KOL Tier-A threshold gets a Breakout badge. Each rising-star row carries the same composite-score drill-down as the KOL ranking; below the table are two sections — Publication trajectory (total-vs-relevant bars) and Score development (the same line chart the KOL profiles use).

Note on the bar charts. Each bar chart now shows two columns per year over 10 years: a light column = the doctor's total publications that year, and a dark inner column = the verified-relevant subset. The light bars are unverified context (a long trend); only the verified-relevant part feeds the score, tiers, tenure and the Rising test. So a tall old total bar with no recent verified-relevant output will not make a doctor a KOL.

The score-development chart

Every KOL profile card — and every rising-star card — carries a small line chart of the composite score year by year, drawn under the publication bars and sharing the same x-axis width, so you can read publication movement and score movement against the same years. The green / blue / grey background bands are the Tier A / B / C score ranges, so you can watch the line climb through the tiers over time; a dashed marker shows the year the doctor crossed out of rising-star tenure.

What this chart is — and isn't. It is a single-run, backward-looking replay against a fixed yardstick: today's final pool is the ruler for every year shown, so the line reflects the individual's own growth, not pool churn. Web sources are a constant baseline (they carry no timestamps), and today's LLM verification verdicts are applied back onto each historical year's PubMed record. It cannot show when a doctor entered or left the KOL pool, or was demoted — that is what the Stage 06 two-run backtest is for. The same year-by-year numbers are exported in the Excel Score by Year sheet.

Thematic distribution (heatmap)

Rows are KOLs, columns are the indication's sub-topics (PCA terms). A cell's shading shows how concentrated that KOL's verified claims are on that theme, relative to the other KOLs in the column — darker = more verified claims on that theme. Empty cell = no verified claim on that theme. It answers "what is each KOL actually known for within the indication?"

Regional distribution

A ranked bar per city, segmented by tier colour (A B C). Bar length reflects the number of KOLs in that city. Useful for territory planning — where the opinion leaders physically cluster.

Collaboration network

An interactive, force-directed graph of who publishes with whom, built from PubMed co-authorship on verified-relevant articles.

  • Blue nodes = KOLs, sized by reach (their distinct-co-author count) — bigger = more collaborators.
  • Purple nodes = external co-authors — frequently-shared collaborators who aren't themselves in the KOL list (e.g. doctors outside our targeting).
  • Lines = shared PubMed papers. Thicker line + a number label = more shared papers. A dashed line means the partner is an external co-author.
  • Only connected KOLs appear on the graph (no lonely dots). Scroll to zoom, drag to pan, hover a node for its institutions or an edge for the shared-paper count.

Two tables below the graph list every edge: PubMed co-authorships and web co-mentions (doctors named alongside each other in verified web statements).

Individual KOL profiles

One card per top KOL: tier badge, composite score and verified-source counts, a 10-year total-vs-relevant publication bar chart above a score-development line chart (same width, so they line up), verified themes, and up to three real verbatim quotes (colour-coded by sentiment, each linking to its source). Every claim on a card survived grounding and adversarial verification — nothing is paraphrased or invented. The per-card "how it was scored" dropdown and the "N years on-topic" sticker were removed to keep the cards uncluttered — the composite is stated on the card, the full weight/norm/contribution breakdown still lives on the KOL ranking rows (and now the rising-star rows), and tenure is readable from the charts.

The Excel workbook

Alongside the HTML, the .xlsx is the analyst's full audit trail — deliberately more detailed than the report, which only shows the top 25. Three sheets:

  • KOLs — one row per doctor: rank, name, specialty, city, tier, rising flag, composite score, verified counts (web / pubmed), latest year, top themes, a representative quote + URL, and the raw scoring internals — norm_relevance / norm_reach / norm_ratio, the three contribution_* values, distinct_coauthors, distinct_affiliations, relevance_ratio, plus total_publications, relevant_tenure, is_kol and breakout.
  • LLM Wiki Verdicts — one row per source handed to the LLM: rank, name, kind (web / pubmed), URL, PMID, a counted / rejected verdict (counted = the source produced ≥1 verified claim), the number of verified claims, and the verified statements / themes / sentiment. This lets you trace exactly why any source did or didn't contribute — the source text itself is not copied in, only the URL and the verdict. (If the underlying sources.json / wiki.json aren't present, the sheet shows a single "not available" note instead.)
  • Score by Year — one row per (doctor, year) from the score-development trajectory: composite score, relevance, reach, ratio, tenure and tier for each year. The same numbers the profile line chart draws, in tabular form for cross-checking.

Questions you should be able to answer

How was the vector search done?

At Stage 01, the indication name and its PCA terms are converted into a 768-number "meaning vector" by a local ONNX embedding model (GTE multilingual, run in-process — no external API). Each candidate web page already has a stored embedding in WEBSITES_VERTICAL_EMBEDDINGS_512. We compute cosine similarity between the query vector and each page; a page becomes a candidate if similarity ≥ 0.55 (vector_sim_threshold), keeping the top 20 per doctor (vector_top_k_per_hcp), still gated by NEAR_BY/IS_DOCTOR. These vector candidates are added to the keyword candidates — a union, not a replacement.

Can the vector search inflate someone's score?

No. The vector arm only widens recall — who gets a chance to be verified. Every candidate it surfaces still has to pass the Stage 03 LLM ground-then-verify pass, which is the sole arbiter of what counts as relevant. A wider net can only let a genuine-but-keyword-invisible doctor earn a score; it cannot manufacture one.

What makes one doctor rank above another?

The composite score. In practice, verified relevance dominates (60% weight), so the doctor with more LLM-verified on-topic sources usually wins; reach (co-author breadth, 25%) and focus ratio (15%) break near-ties and reward broad, topic-specialist doctors. Open any row's breakdown to see the exact per-factor contributions.

Why isn't specialty used to filter?

Because hardcoding specialty exclusions is how v1 went wrong in the other direction and is brittle. Instead, relevance verification does the filtering: an off-target specialist simply won't accumulate verified on-topic sources and falls out during the 100→25 buffer. Specialty is displayed for context only.

Why are web sources not date-filtered like PubMed?

Web records in LLM_VALIDATION are timestamp-free; the only freshness signal is IS_OLD=0, which we apply. PubMed articles do have years, so we window them (10 years) and can pin them to a past anchor year for backtesting. In a backtest run, web sources are shown as-is (frozen across years) and the report flags this.

How do we know the ranking is any good, with no "correct answer" to compare to?

Via the Stage 06 backtest. We run the pipeline pinned to a past year and again on the latest data, then check whether the doctors we'd have flagged as rising stars back then actually became today's KOLs. Promotions, tier moves and new KOLs are written to backtest_compare.json.

Where do the verbatim quotes come from — could they be made up?

No. Each quote is extracted by the ingest LLM, then grounded — it must appear as a literal substring of the actual source text, or it's dropped before any further processing — and then the claim is adversarially verified by a second model. Only quotes that survive both appear in the report, each linked to its source.

The one-line summary

A KOL's position is earned, one verified source at a time, then fine-tuned by how widely they collaborate and how focused they are — every number in the report traces back to specific evidence you can click through to.
