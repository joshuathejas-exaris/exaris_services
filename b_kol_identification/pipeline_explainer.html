Service 2.1 · KOL Identification & Mapping

How the pipeline works

A step-by-step walkthrough: how we rank the true opinion leaders for an indication by counting the genuinely relevant sources attributed to each doctor — and why we spend LLM budget only on the handful of candidates who could actually make the top list.

In one sentence

For an indication (e.g. Obesity), we count — per doctor — how many sources genuinely show that doctor engaging with the indication (web statements they actually made, and PubMed articles they actually authored on-topic). The doctors with the most verified relevant sources are the KOLs. Everything that is merely a keyword co-occurrence is thrown away.

Why we rebuilt it

The first version scored HCPs with 0.45·pub + 0.30·cf + 0.25·digi and produced professionally wrong rankings — human geneticists and lab-medicine physicians landed in the Obesity top-50. Two failure modes caused it, and the whole redesign exists to kill them:

1 · Co-occurrence ≠ engagement

The old cf_count only meant an indication keyword and a doctor's name appeared in the same document. It never checked whether the doctor said anything about the topic.

2 · Wrong signals dominated

DigiScore (25% weight) floated doctors with ≤1 publication to the top, and PubMed "relevance" was a global similarity cut with no per-doctor topical check.

The one principle

A KOL's score = the number of genuinely relevant sources attributed to them. "Genuine" is decided the same way on both tracks: an LLM confirms the doctor is actively engaging with / sharing a view on the indication in that source. No DigiScore. No raw keyword counts. No global similarity percentile. This encodes the client's own criteria: most relevant sources and recency.

The cost funnel — cheap first, LLM last

We must not send thousands of documents to the LLM. So we narrow the field with cheap SQL first, and only the survivors ever reach the expensive step. This is the heart of the design:

ALL HCPs (~1,200)Stage 01 · cheap SQL only — no LLM · count candidate web + PubMed sources per doctor
▼
Top ~75 candidatesranked by candidate-source count (an upper bound on the real score) · everyone else dropped before any LLM spend
▼
Fetch full textStages 02 · web CONTENT + PubMed title/abstract for the 75 · no vector search
▼
LLM verifiesStage 03 · ingest → ground → verify · a source counts only if a claim survives both checks
▼
Top 25 KOLsStages 04–05 · rank on verified sources → report

The 75 → 25 buffer absorbs candidates that look strong on cheap counts but collapse under verification — exactly the off-target-specialty problem. They simply fall out; a genuine doctor takes the slot. No hardcoded specialty exclusions — relevance verification does the filtering.

Two evidence tracks

Every source is one of two kinds. Both are counted the same way in the end, but they are gathered and verified slightly differently:

Web / practice communication

From the LLM_VALIDATION table. Here the doctor's authorship is unknown, so attribution is the whole problem — the LLM must confirm the named doctor is actively discussing the indication, not merely co-present on the page.

Cheap gate: NEAR_BY=1, IS_OLD=0, IS_DOCTOR=1, IN_RELATION>29, keyword ∈ indication terms.

PubMed science

From PUBMED_ARTICLE_MAPPING (merge_result>1 = verified author) → text from CORE.PUBMED.ARTICLE. Authorship is already proven, so the LLM's job is relevance: is this article genuinely about the indication? Co-authors (incl. doctors outside our targeting) come structurally from CORE.PUBMED.AUTHOR.

The five stages

Each stage writes a JSON checkpoint and is resume-safe (skips if its output exists, unless you pass --force). They run in order:

01

Fetch & shortlist

01_fetch_and_shortlist.py
What happens (cheap SQL, no LLM):
  • Load the indication's terms — CONTENT_FRAME_SPEC where PCA='X' (configurable to all terms).
  • Web candidates per doctor: gate LLM_VALIDATION on NEAR_BY=1, IS_OLD=0, IS_DOCTOR=1 + IN_RELATION>29 + a term match → candidate WEBSITE_IDs + a count. (IS_OLD=0 is the only freshness filter — no date window on web.)
  • PubMed candidates per doctor: verified authorship (merge_result>1) ∩ a topical CF-flag, within a 5-year window → candidate PMIDs + a CF-treffer-weighted count. The window is anchored on MAX(YEAR_VAL) in PUBMED_CONTENT_FRAME_SINGLE_TBL (the latest year the targeting actually covers — e.g. 2023), not the current calendar year, so a 2023 targeting scores 2018–2023.
  • Publication-history bars (display only): a separate query counts each doctor's CF-flagged PubMed articles per year over 20 years back from the same anchor. These counts drive the bar charts only — they are never sent to the LLM and never affect the score. Bar height = CF-flag count (keyword-level activity, unverified); only the most-recent-5-years articles go through LLM-wiki verification and contribute to the score, tier, and rising-star flags.
  • Shortlist: candidate_score = web_count + pubmed_count (an upper bound on the final verified score) → keep the top ~75. Metadata (name, city, specialty, A/B/C/D rating) is attached; specialty is displayed, never used to filter.
CONTENT_FRAME_SPEC LLM_VALIDATION PUBMED_ARTICLE_MAPPING PUBMED_CONTENT_FRAME_SINGLE_TBL CUSTOMER_SOURCE + RATING_RESULT_FINAL no LLM
→ data/shortlist.json
02

Retrieve source text

02_retrieve_sources.py
What happens (cheap SQL, no vector search): for the ~75 shortlisted doctors only, pull the actual text of every candidate source:
  • Web: full CONTENT from LLM_VALIDATION for each candidate WEBSITE_ID.
  • PubMed: TITLE + ABSTRACT from CORE.PUBMED.ARTICLE for each candidate PMID.
Why no vector search? Service 1.2 used it to pick on-topic docs out of the whole corpus. Here the funnel (75 doctors) and the gate (IN_RELATION>29 + term match) already guarantee on-topic docs — so we simply take all of them. A recency-ordered per-doctor cap is the only limiter, purely to bound cost.
LLM_VALIDATION (CONTENT) CORE.PUBMED.ARTICLE no LLM · no embeddings
→ data/sources.json
03

LLM wiki-build — where relevance is verified

03_wiki_build.py
What happens, per source (web & PubMed alike):
  • Ingest (LLM) — extract whether the doctor is genuinely engaging with the indication: a verbatim supporting quote, a one-line statement, sentiment, the themes (which indication terms), and any other doctors mentioned.
  • Quote-grounding (deterministic) — the verbatim quote must literally appear in the source text, or the claim is dropped for free — before spending a verify call.
  • Verify (a second, different LLM, adversarial) — returns true only if the source really shows this doctor engaging with the indication.
  • Map — resolve the doctor and any mentioned doctors against our targeting roster by name.
A source counts as relevant for a doctor only if it yields ≥1 grounded + verified claim. Writes a browsable wiki/<ts>/ tree with a drops.json recording everything discarded and why.
Ingest: Nova Pro (Bedrock) Verify: Qwen3-235B (Bedrock) no Snowflake — works on Stage-02 output
→ data/wiki.json  (+ wiki/<ts>/ tree)
04

Assemble the KOL ranking

04_assemble_kols.py
What happens (no LLM):
  • Final score = verified web sources + verified PubMed sources. Rank descending; latest source year breaks ties (recency).
  • Tiers A B C from the score distribution of the shortlist.
  • Rising Stars — recent vs. prior activity (same logic as before), but computed only on verified-relevant PubMed articles by year, so we never crown a rising star on off-topic output.
  • Collaboration network — PubMed co-authors (structural, incl. non-targeted doctors) + web co-mentions (from the LLM). Thematic distribution — aggregated from each doctor's verified claim themes.
CORE.PUBMED.AUTHOR no LLM — plain tallies
→ data/kol_final.json
05

Generate the report

05_generate_report.py
What happens: renders a self-contained HTML report (top 25) — executive dashboard, KOL ranking, rising stars, thematic distribution, regional distribution, collaboration network, and individual profiles with real verbatim quotes + source links — plus an Excel export with HCP-level rows (name, institution, specialty, verified-source count, themes, source URLs, quotes). Professional multi-hue palette, not the old monotone blue.
no LLM — pure rendering

Stages 03–05 don't touch Snowflake; they work purely on the JSON checkpoints. The pipeline reads across three databases (CUST_NOVO, CUST_TC, CORE) in one connection via fully-qualified names.

To confirm on the first live run: the CF-flag column layout in PUBMED_CONTENT_FRAME_SINGLE_TBL; that IN_RELATION is numeric so >29 is the right cut; that IS_OLD=0 is the intended web freshness filter; and the fully-qualified locations of the cross-database tables. We also log per-doctor candidate counts for the top 75 before Stage 03, to sanity-check LLM volume.

How we get the final result

End to end

  1. Count candidates cheaply — web gate + verified PubMed authorship, per doctor (Stage 01).
  2. Shortlist the top ~75 and fetch their source text — no LLM yet (Stages 01–02).
  3. Verify relevance with the LLM — grounded, adversarially confirmed; a source counts only if it survives (Stage 03).
  4. Rank on verified relevant sources — tiers, rising stars, network, themes (Stage 04).
  5. Report the top 25 — HTML + Excel, every KOL backed by real quotes and source links (Stage 05).

The result is a defensible KOL list: each ranked doctor is there because a specific number of sources prove they engage with the indication — with co-occurrences, wrong specialties, and unsupported signals filtered out along the way.

Under the hood

  • LLMs: AWS Bedrock (eu-central-1) — Amazon Nova Pro for ingest, Qwen3-235B for the adversarial verify. Temperature 0. Reused from Service 1.2's pipeline_common.call_bedrock_json.
  • No embeddings / no reranker: the funnel + SQL gate replace vector search entirely.
  • Snowflake auth: AWS Secrets Manager private-key auth via the shared ParameterManager / SecretReader helpers.
  • Cost control: only ~75 doctors' sources within the anchored 5-year window reach the LLM; the 20-year history is cheap SQL counts for the charts and never hits the LLM. Grounding drops fabricated quotes before any verify call is spent.
