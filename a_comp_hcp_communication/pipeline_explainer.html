Service 1.2 · Competitive HCP Communication Monitoring

How the pipeline works

A step-by-step walkthrough for colleagues: what each stage does, which Snowflake tables it reads, and how we get from a client drug to a traceable report of what doctors actually say about competitor drugs.

In one sentence

Given a client drug (e.g. Ozempic), the pipeline finds its competitors, pulls real source documents where doctors discuss them, and extracts what each doctor actually said — attributing a statement to a doctor only when the source text proves it — then produces an HTML + Excel report.

Why it's built this way

The whole design exists to kill two failure modes. Everything downstream — the grounding, the double-check — is there to prevent these:

1 · False attribution

A doctor is named on a page, the drug is mentioned somewhere else on the same page, and we wrongly credit the doctor with an opinion they never expressed.

2 · Co-occurrence ≠ opinion

A keyword matching in a document is not proof of a view. Being in the same text as "Mounjaro" doesn't mean the doctor said anything about it.

So every claim is grounded twice: (a) the verbatim quote must literally appear in the source text (a deterministic string check), and (b) an independent LLM "verify" pass confirms the named doctor really expresses that view about that drug. Anything that fails either check is dropped.

Two tracks & two doctor types

Every competitor is processed on one of two tracks, decided back in Stage 01:

Track A — CF-derived (source = "cf")

The competitor came from our content-frame terms. The LLM_VALIDATION table is meaningful here, so we can apply the full gate and get mapped HCPs (real records with an S_CUSTOMER_ID).

Track B — LLM-knowledge (source = "llm")

The competitor came from the model's own knowledge (CF gave too few). No LLM_VALIDATION rows — we do a whole-corpus vector search only, and every doctor found is unmapped by construction.

Mapped vs. unmapped: a mapped doctor resolves to a customer record (S_CUSTOMER_ID); an unmapped doctor is genuinely quoted in a source but has no matching record. Both are reported — each clearly flagged.

The five stages

Each stage writes a JSON checkpoint and is resume-safe (it skips if its output already exists, unless you pass --force). They run in order:

01

Identify competitors

01_identify_competitors.py  ·  python 01… --client-drug "Ozempic" --from-snowflake
What happens: Reads content-frame (CF) terms for the indication. Row 0 is treated as the indication; the rest are candidate keywords. An LLM maps the client drug + CF terms to concrete competitor drugs (brand + generic name). If fewer than 2 competitors come from CF terms, the LLM adds more from its own knowledge. Each competitor is tagged source: "cf" (→ Track A) or "llm" (→ Track B). It also rejects a drug name that accidentally leaked in as the indication.
CONTENT_FRAME_SPEC → DE_TERM_1, EN_TERM_1 LLM: Qwen3-235B (Bedrock)
→ data/competitors.json
02

Retrieve sources

02_retrieve_sources.py
What happens (the 2-layer retrieval):
  • Layer 1 — the gate (Track A only): query LLM_VALIDATION for rows where IS_DOCTOR = 1 AND IS_OLD = 0 AND NEAR_BY = 1 AND the brand/generic appears in COL_KEYWORDS_ORIG / COL_KEYWORDS_EN. (IN_RELATION exists but is intentionally not gated — it's non-indicative.) This yields the relevant WEBSITE_IDs and the mapped-HCP roster.
  • Layer 2 — vector search: embed the query strings (brand, generic, "brand + indication") locally and run VECTOR_COSINE_SIMILARITY against pre-computed 512-token chunk embeddings — restricted to the Layer-1 websites (Track A) or the whole corpus incl. PubMed (Track B). Keep the best 100 chunks/drug at similarity ≥ 0.65, capped at 40 sources/competitor.
The chunks decide which documents matter; the full document CONTENT of those websites becomes the raw text for the wiki (truncated to 24k chars).
LLM_VALIDATION WEBSITES_VERTICAL_EMBEDDINGS_512 WEBSITES_EMBEDDINGS_512 PUBMED_EMBEDDINGS_512 CUSTOMER_SOURCE local ONNX embedder (GTE multilingual, 768-dim)
→ data/raw_sources.json
03

Build the LLM-wiki (this is where grounding happens)

03_wiki_build.py
What happens, per source document:
  • Ingest (LLM) — extract only concrete statements a named doctor makes about the specific drug: (speaker, verbatim quote, paraphrased statement, sentiment, confidence). Financial-disclosure / conflict-of-interest text is excluded.
  • Quote-grounding (deterministic) — check the verbatim quote literally appears in the source text. If not → dropped (grounding). This runs before spending an LLM verify call.
  • Verify (LLM, adversarial) — a second, independent pass returns true only if the quote is present AND the document shows this speaker expressing this view. Fail → dropped (verify).
  • Speaker mapping — match the speaker name to the mapped-HCP roster from Stage 02 → attach S_CUSTOMER_ID, or flag "not mapped".
Writes a browsable wiki/<timestamp>/<competitor>/ tree (raw/ source pages, wiki/ one page per doctor, schema/ the graph + a drops.json listing everything discarded and why). Sentiment (positive / negative / ambivalent / neutral) is an attribute of each surviving claim.
Ingest: Nova Pro (Bedrock) Verify: Qwen3-235B (Bedrock) no Snowflake — works on Stage-02 output
→ data/knowledge_graph.json  (+ wiki/<ts>/ tree)
04

Synthesize

04_synthesize.py
What happens: takes the already-verified claims (no re-judging) and aggregates them. Per competitor it counts a sentiment split across mapped / unmapped / all doctors, then an LLM writes a short market-view narrative (2–4 sentences, grounded strictly in the listed statements). One overall executive summary is written across all competitors.
LLM: Qwen3-235B (Bedrock) counts are plain tallies — no weighting/thresholds
→ data/synthesis.json
05

Generate the report

05_generate_report.py
What happens: renders the final deliverables. The primary report is tabbed — overview, per-competitor, doctors-weighing-in, most-active doctors, and a methodology section — showing ~15 examples per section with an "N more" roll-up. Every claim links back to its source. Conflict-of-interest disclosures are filtered out of both HTML and Excel.
no LLM — pure rendering
→ results/report_<ts>.html · guide_<ts>.html · technical_<ts>.html · report_<ts>.xlsx

If that ever turns out to be a fragment rather than the whole document, switch [wiki] content_source = chunk_concat in config.ini to reconstruct text by concatenating the *_EMBEDDINGS_512.CHUNK rows per WEBSITE_ID.

How we get the final result

End to end

  1. Competitors from the client drug + CF terms (Stage 01).
  2. Trusted source documents via the LLM_VALIDATION gate + vector search (Stage 02).
  3. Grounded, verified claims — quote-checked and adversarially confirmed, each tied to a (possibly mapped) doctor (Stage 03).
  4. Aggregated view — sentiment splits + narrative per competitor (Stage 04).
  5. Reports — tabbed HTML + full Excel, every statement traceable to its source (Stage 05).

The result is a list of competitor drugs, and for each: which doctors are talking about it, exactly what they said (verbatim + source link), the sentiment, and whether that doctor is a known (mapped) HCP — with false attributions and mere co-occurrences filtered out along the way.

Under the hood

  • LLMs: AWS Bedrock (region eu-central-1) — Qwen3-235B for competitor ID, verify & synthesis; Amazon Nova Pro for claim extraction. All at temperature 0.
  • Embeddings: a local ONNX model (GTE multilingual, 768-dim) — queries embedded at runtime, chunk embeddings pre-computed in Snowflake; similarity via VECTOR_COSINE_SIMILARITY.
  • Snowflake auth: AWS Secrets Manager private-key (key-pair) auth via the shared ParameterManager / SecretReader helpers.
  • Reranker: reranker.py exists but is unused after the rework (kept for parity).
